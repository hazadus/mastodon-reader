You can let users of your library take advantage of the bleeding edge if they so wish without cutting off one of the currently supported Go toolchains.

When something tantalising gets merged at tip, you can put code that relies on it in a file guarded by a //go:build go1.27 build constraint and its pre-1.27 counterpart in a file guarded by a //go:build !go1.27 build constraint. No need to update your go.mod's go directive.

**🕒 18:34 👤 @itamarst@hachyderm.io**
💬 The basic idea is that you have a set of complexity models, and set of inputs. You run the function being tested under the inputs, and now you have a set of points `(input size, elapsed time of function(input))`. You fit that set of points to all the models; if the model with the best fit is yours, congrat. If not, it panics.

Model fitting is done with Levenberg-Marquardt, fitting parameters A and B of `A + B * model(n)` to the `(input size, measured complexity of function(input))` where `n` is the input size. The least squares difference is what is being minimized. Best fit is the model with the lowest Akaike Information Score, calculated from the least squares difference from the real data points. This part is all directly based on [https://github.com/plasma-umass/bigO/](https://github.com/plasma-umass/bigO/)

Measuring elapsed time tries to use CPU instruction counts when possible on Linux, falling back to the thread's CPU time if it's not possible (GitHub Actions being one notable example).

Getting CPU instruction counts to work at all reliably is a pain, especially since this going to run in an environment I don't control (`cargo test` thread pool). Sometimes it just return unexpectedly low numbers. Sometimes it captures CPU events for just part of the function run. Sometimes it doesn't work at all. I just gave up on pinning threads to a core, it breaks perf_event_open() completely. So I do a bunch of annoying things:

1. Run up to 100 iterations, trying to measure up to ~20 full duration measurements.
2. If I get a measurement of 0, meaning "I'm sorry, Dave, I'm afraid I can't do that" sleep for a while in the hopes of switching to a core/kernel clearing whatever resource management thing it's getting wrong(?!).
3. Choose the _maximum_ measurement cause, again, sometimes measurements are way lower for no clear reason.
